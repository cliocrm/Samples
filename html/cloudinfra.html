  <h3 style="font-size:3.5vw;color:white;">Remote Storage Replication in SaaS DR and other use cases</h3>


<h3>

Replication of data is important to maintain the integrity and consistency of data and assets of a SaaS application in the cloud. Since storage devices can be present in datacenters across the globe, it should be feasible to control and administer the replication process from anywhere using a suitable interface. Typically all the administration happens via CLI or BUI (browser user interface).

<br/>

With reference to the replication procedures discussed in the product documents for <a href="https://www.oracle.com/technetwork/server-storage/sun-unified-storage/documentation/zfssa-replication-2014-1-2120969.pdf">Oracle ZFS storage appliance</a>, I will be discussing a few core concepts around remote replication for disaster recovery and other use cases. I was familiar with the previously known Sun Storage 7000 appliances but you can refer to the above-mentioned manual for the latest developments in this. The ZFS appliance is a complex hardware and is the best in class and good to get familiar with replication concepts because of the extensive documentation maintained for years since its use on-premise. Below is a brief description of the use of storage in SaaS products we use, hosted in the public or private cloud.<br/>

The provisioning procedure uses frontend and backend IP addresses. The frontend IPs are for receiving the client (public-facing) requests. Once it reaches the cloud infrastructure, processes are initiated for completing the provisioning operations with the help of the provisioning service and generating the notifications with the role-based credentials for the tenant created in IdM for the SaaS product, and once primary resources like IP addresses, RAM and storage (hard disk) requirements for the VMs are determined, the backend IPs are used to communicate with the ZFS storage appliances to create the storage footprint of the VMs created. The NFS protocol comes into play at this time. Applications like enterprise CRM, HCM, ERP, etc.  have a huge requirement in terms of storage as compared to SDK or other lighter cloud SaaS applications because of the number of modules that they are composed of. It is possible to SSH into each of the ZFS appliances via the provisioning software to create a share or a project and the file system utilities via connectors. Scripts can be run for various operations needed that contain the ZFS commands corresponding to a particular action like zfs create datapool/fs1 for creating the filesystem fs1 under pool datapool. <br/>

There are load balancers for both the internal and external IP addresses to distribute the traffic uniformly choosing the best storage appliance that will complete the service request without failure using the allocation algorithm.<br/>

Given this background, we can now visualize the role of replication of the storage system in disaster recovery. The filesystem contains all the data the tenant has for the service like CRM and it needs to have a backup in case there is an outage so that the data is not lost.<br/>

Other scenarios for the need for remote replication are:<br/>

1. Data migration<br/>

It can be done via Shadow migration also to upgrading hardware or rebalancing storage.<br/>

2. Data distribution<br/>

To distribute media or VM images to client systems that are unable to reach their own systems due to latency or other issues.<br/>

3. Data backup<br/>

To replace tape backups.<br/>

<br/>

The main properties of the process include snapshot creation and update on a change basis if repeated later. It is asynchronous in sending the continuous streams of data to the SAN and NAS clients. The metadata related to storage and user data is serialized and sent over NFS to a host of storage servers. This communication over the network is secured using SSL or other authentication techniques. The replication process can be manual, continuous or scheduled and it can happen either at the Share level or the Project level. <br/>

An important concept here wrt DR is reverse replication for two-way replication. Along with that many other enhanced modes have evolved as mentioned in the document. This feature is typically used to implement a two-system disaster recovery configuration in which a primary system serves production data and replicates it to a secondary or DR system (often in another data center) standing by to take over the production traffic in the event of a disaster at the primary site. In the event of a disaster at the primary site, the secondary site's copy must be made "primary" by making it writable and redirecting production traffic to the secondary site. When the primary site is repaired, the changes accumulated at the secondary site can be replicated back to the primary site and that site can resume servicing the production workload.
