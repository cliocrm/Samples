Cross-browser multi-device compatibility & accessibility in HTML

Web development is common like never before, be it a simple website with static data or a web application that accepts user inputs and generates an HTML on the fly with the required backend operations on the data. While adding the text and images with a complex UI design, we assume that the HTML with the CSS will appear as expected on a commonly used browser like Chrome or Safari or MS Edge on a mobile device or any larger screen like a Linux laptop or an iPad. Developers and testers often ignore checking the presentation of the text and visual data in different browsers across different devices. The consequences are that it frustrates the end-users if they are unable to see the images or text or other widgets properly on the UI.
From whatever I have seen in the past it is important to set some minimum requirements for any HTML-based deliverable to be tested for compatibility across all available browsers and devices. Meaning if we have five browsers in total at present in the market say: Chrome, Firefox, Edge, Safari, and Chromium, and 10 devices with some standard operating systems: Macbook with Sierra, Dell laptop with Ubuntu, Dell laptop with Windows 10, Galaxy note s10 with Android Oreo, Google Pixel with Android Marshmallow, iPhone 11, iPhone 8, iPad, Microsoft Surface, Nokia with Windows Mobile, Samsung Note 10 lite, then we will have a suite of tests to be run for 5*10=50 combinations. This can be either as a manual exercise or an automated suite depending on the expertise of the team. But the best way is to test it during the development phase itself to avoid making changes to the HTML files later. The code for the test suite can always be reused for different projects with required changes across the organization.
Accessibility is another aspect quite old because it is always important for the products to be inclusive and be designed that way. There are tools like JAWS that guide developers to test for the corresponding audio version of the HTML for being understandable to the visually impaired. Mails generated as a part of a notification service of an enterprise application should especially take care of this because it is mandatory for the emails or simple web pages to be easily converted into a paragraph and be read out without any difficulty in case the end-user wants to listen to the content instead of reading it. It might involve interpreting the visually represented data like a graph or a chart to be converted to text in a human-understandable way such that the user can make sense. This is somewhat like a natural language generation and understanding problem. Long passages, images, charts or tables cannot be easily converted to a paragraph description without understanding the context and relation between the different elements there. For example, an analysis of the change in the demographics for the current year may contain a set of line graphs that are related apart from the text description. To establish a relationship, we need prior knowledge of the background of the use case. Without that, we will not be able to read out the interpretation well. Similarly for medical images with some data about the tests and findings, need to be constructed with medical context.
One way is to use the "alt text" placeholder in an HTML for describing a section in the HTML, but what goes in there again depends on how we define it. We might either want to hardcode that part with manually entered text or generate it with the help of an algorithm. The second option is more feasible when we are working on a large set of web pages or emails being sent out, which is usually the case. Whether or not we choose to use NLP, computer vision or other latest technologies is our choice.
